# iidy Rust Implementation Plan
## Analysis of Missing Features and UX Polish

Based on analysis of the original iidy-js implementation (from the codebase, documentation, and [demo video](https://asciinema.org/a/8rzW1WyoDxMdVJpvpYf2mHm8E)), this document outlines what functionality and UX improvements are needed to achieve feature parity and production readiness.

## Current State Summary

✅ **Completed Major Features**:
- Core CloudFormation operations (create, update, delete, watch, describe)
- Comprehensive token management system with idempotency 
- Changeset workflows with interactive confirmation
- Stack drift detection with formatted output
- Demo command with YAML scripting
- Basic YAML preprocessing framework
- Offline testing architecture
- CLI argument parsing with clap

❌ **Missing Critical Features**:
- Complete YAML preprocessing language implementation
- SSM Parameter Store integration (`iidy param` commands)
- Template approval workflow (`iidy template-approval` commands) 
- Rich stack metadata display and formatting
- Advanced import system (s3, cfn:export, git, etc.)
- Professional UX with proper colors, formatting, and metadata display

## Gap Analysis by Category

### 1. CLI Commands and Functionality

#### Missing Commands:
- **`iidy param`** - SSM Parameter Store management
  - `param set <path> <value>` - Set parameter value
  - `param get <path>` - Get parameter value  
  - `param get-by-path <path>` - Get multiple parameters
  - `param get-history <path>` - Parameter history
  - `param review <path>` - Review pending changes

- **`iidy template-approval`** - Template approval workflow
  - `template-approval request <argsfile>` - Request approval
  - `template-approval review <url>` - Review approval request

- **`iidy render <template>`** - Standalone YAML preprocessing
- **`iidy get-import <import>`** - Test import sources directly
- **`iidy lint-template <argsfile>`** - Template validation
- **`iidy convert-stack-to-iidy <stackname> <outputDir>`** - Reverse engineering
- **`iidy init-stack-args`** - Project initialization

#### Missing CLI Features:
- **Environment-based configuration** (`--environment` flag working with stack-args.yaml)
- **Stack metadata display** (region, profile, IAM principal, tags)
- **Better help formatting** (colors, grouping, descriptions)
- **Bash completion** generation
- **`--color` option** (auto/always/never)

### 2. YAML Preprocessing Language

#### Current State:
- Basic framework exists (`src/yaml/`)
- Handlebars engine partially implemented
- Import system started but incomplete

#### Missing Core Features:
- **Data Import Sources**:
  - ✅ `file` (basic implementation exists)
  - ❌ `s3://bucket/path` - S3 object loading
  - ❌ `cfn:export:name` - CloudFormation exports
  - ❌ `cfn:output:stack/output` - Stack outputs
  - ❌ `cfn:parameter:stack/param` - Stack parameters
  - ❌ `cfn:tag:stack/tag` - Stack tags
  - ❌ `cfn:resource:stack/resource` - Stack resources
  - ❌ `cfn:stack:stack` - Complete stack data
  - ❌ `env:VAR` - Environment variables
  - ❌ `git:branch`, `git:describe`, `git:sha` - Git information
  - ❌ `random:dashed-name`, `random:int` - Random values
  - ❌ `ssm:/path` - Parameter Store integration
  - ❌ `http://url` - HTTP/HTTPS loading

- **Custom YAML Tags**:
  - ❌ `!$if { test, then, else }` - Conditional logic
  - ❌ `!$eq [a, b]`, `!$not bool` - Comparison operators
  - ❌ `!$map { items, template, var }` - List transformation
  - ❌ `!$merge [maps]` - Object merging
  - ❌ `!$concat [lists]` - Array concatenation
  - ❌ `!$split [delim, string]` - String splitting
  - ❌ `!$join [delim, array]` - Array joining
  - ❌ `!$let { var: val, in: template }` - Variable binding
  - ❌ `!$escape content` - Preprocessing escape
  - ❌ `!$parseYaml`, `!$parseJson` - Format parsing

- **Handlebars Helpers**:
  - ❌ `toJson`, `toYaml` - Format conversion
  - ❌ `toLowerCase`, `toUpperCase` - Case conversion
  - ❌ `base64` - Encoding
  - ❌ String manipulation helpers from handlebars-helpers

## Implementation Plan

### Phase 1: Core YAML Preprocessing
**Priority: Critical** - This is the foundational feature that enables advanced template composition

#### 1.1 Import System Enhancement
- Complete import loader implementations
  - Finish `env`, `git:*`, `random:*` loaders  
  - Implement `http`/`https` loading with proper error handling
  - Add `cfn:*` loaders for CloudFormation integration
  - Implement `ssm` and `ssm-path` loaders

#### 1.2 Custom YAML Tags
- Implement core logical tags
  - `!$if`, `!$eq`, `!$not` for conditional logic
  - `!$map`, `!$merge`, `!$concat` for data transformation
  - `!$split`, `!$join` for string/array manipulation
  - `!$let` for variable binding

#### 1.3 Handlebars Enhancement  
- Complete handlebars integration
  - Add all string helpers (`toJson`, `toYaml`, `base64`, etc.)
  - Integrate handlebars-helpers equivalent functionality

## Risk Assessment and Mitigation

### High Risk Areas:
1. **YAML Preprocessing Complexity**: Custom tag system is complex
   - *Mitigation*: Extensive testing, gradual implementation, reference JS behavior
   
2. **AWS Integration Breadth**: Many AWS services to integrate 
   - *Mitigation*: Leverage existing AWS SDK patterns, offline testing

3. **UX Consistency**: Matching professional CLI experience
   - *Mitigation*: Study demo video carefully, consistent design system

### Dependencies and Blockers:
- Phases 2+ depend on Phase 1 YAML preprocessing
- UX enhancement (Phase 2) can be done in parallel with Phase 1
- Template system (Phase 3) depends on Phase 1 completion
- SSM integration (Phase 6) can be done independently after core features

## Implementation Approach

**Recommended sequencing**: Execute Phase 1 first, then parallelize Phases 2-3, followed by remaining phases based on priorities.

- **Phase 1** (Core YAML): *Critical path* - Foundation for all advanced features
- **Phase 2** (UX): *High priority, can parallelize with Phase 1* - Professional user experience
- **Phase 3** (Templates): *High priority* - Advanced CloudFormation workflows
- **Phase 4** (Approval): *Medium priority* - Enterprise security feature
- **Phase 5** (Polish): *Low priority* - Quality of life improvements
- **Phase 6** (SSM): *Low priority* - Can be added after core functionality is complete

This plan transforms the current functional Rust implementation into a production-ready CloudFormation tool with complete feature parity to the original iidy-js, while maintaining the architectural improvements (better error handling, type safety, token management) already achieved.
