# Token Management Design and Implementation Plan

## Overview

This document outlines the design and implementation plan for comprehensive client request token management in the Rust iidy CloudFormation tool, supporting both single-step and multi-step operations with full visibility and testability.

## Current State Analysis

### What We Have
- ✅ `--client-request-token` CLI option in `AwsOpts` as `Option<String>`
- ✅ `CfnContext` with timing abstraction for reliable event filtering
- ✅ Basic CloudFormation operations (some still stubs)
- ✅ `StackArgs` support for CloudFormation configuration
- ✅ **TokenInfo and derivation system implemented** (Phase 1, Commit 1)
- ✅ **NormalizedAwsOpts with guaranteed TokenInfo presence** (Phase 1, Commit 2)
- ✅ **Token generation and normalization logic** (Phase 1, Commit 2)
- ✅ **Comprehensive test coverage for token functionality** (Phase 1, Commit 2)

### What's Missing
- ❌ CfnRequestBuilder pattern for standardized API calls
- ❌ Multi-step operation support (changeset workflows)
- ❌ Console visibility of tokens being used
- ❌ Comprehensive offline testing with fixtures

## Design Requirements

### Token Management Requirements
1. **Token Generation**: Auto-generate UUID if not provided via CLI
2. **Token Derivation**: Create deterministic sub-tokens for multi-step operations
3. **Visibility**: Always show what tokens are being used with source indication
4. **Consistency**: All CloudFormation operations that support idempotency should use tokens
5. **Traceability**: Full audit trail of token usage across operation steps

### Multi-Step Operation Requirements
- `iidy update-stack --changeset` needs separate tokens for create-changeset and execute-changeset
- `iidy create-or-update --changeset` needs tokens for stack existence check, changeset creation, and execution
- Each step must be independently retryable with same derived token
- Handle AWS API inconsistencies (changeset uses `client_token`, others use `client_request_token`)

### Testing Requirements
- ✅ **Fully offline testing** with no network dependencies
- ✅ **Deterministic behavior** with fixture-based responses
- ✅ **Reproducible CI/CD** that doesn't depend on external services
- ✅ **Fast test execution** without network latency

## Token Management Strategy

### TokenInfo Structure
```rust
#[derive(Debug, Clone)]
pub struct TokenInfo {
    pub value: String,
    pub source: TokenSource,
    pub operation_id: String, // For correlation
}

#[derive(Debug, Clone)]
pub enum TokenSource {
    UserProvided,
    AutoGenerated,
    Derived { from: String, step: String },
}
```

### Token Derivation Algorithm
- Use deterministic hashing: `SHA256(original_token + step_name)`
- Format: `{first_8_chars_of_original}-{first_8_chars_of_hash}`
- Example: `abc123ef-a7b2c8d4` derived from `abc123ef-4567-89ab-cdef-0123456789ab` for step "create-changeset"

### NormalizedAwsOpts
```rust
pub struct NormalizedAwsOpts {
    pub region: String,
    pub client_request_token: TokenInfo, // Always present after normalization
    pub fixture_set: Option<String>,      // For testing
}

impl AwsOpts {
    pub fn normalize(self) -> NormalizedAwsOpts {
        let token_info = match self.client_request_token {
            Some(user_token) => TokenInfo {
                value: user_token,
                source: TokenSource::UserProvided,
                operation_id: uuid::Uuid::new_v4().to_string(),
            },
            None => TokenInfo {
                value: uuid::Uuid::new_v4().to_string(),
                source: TokenSource::AutoGenerated,
                operation_id: uuid::Uuid::new_v4().to_string(),
            },
        };
        
        NormalizedAwsOpts {
            region: self.region,
            client_request_token: token_info,
            fixture_set: self.fixture_set,
        }
    }
}
```

## Architecture Components

### Enhanced CfnContext
```rust
pub struct CfnContext {
    pub client: Client,
    pub time_provider: Arc<dyn TimeProvider>,
    pub start_time: Option<DateTime<Utc>>,
    pub token_info: TokenInfo,  // Primary token info
    pub used_tokens: Arc<Mutex<Vec<TokenInfo>>>, // Track all tokens used
}

impl CfnContext {
    pub fn derive_token_for_step(&self, step: &str) -> TokenInfo {
        let derived = self.token_info.derive_for_step(step);
        
        // Track the derived token
        if let Ok(mut used) = self.used_tokens.lock() {
            used.push(derived.clone());
        }
        
        derived
    }
}
```

### CfnRequestBuilder Pattern
```rust
pub struct CfnRequestBuilder<'a> {
    context: &'a CfnContext,
    stack_args: &'a StackArgs,
}

impl<'a> CfnRequestBuilder<'a> {
    pub fn build_create_stack(&self, step_name: &str) -> (CreateStackInput, TokenInfo) {
        let token = self.context.derive_token_for_step(step_name);
        
        let mut builder = self.context.client.create_stack()
            .stack_name(self.stack_args.stack_name.as_ref().unwrap())
            .client_request_token(&token.value);
            
        // Apply StackArgs configuration
        if let Some(ref template) = self.stack_args.template {
            builder = builder.template_body(template);
        }
        
        // Apply other StackArgs fields (capabilities, parameters, tags, etc.)
        
        (builder, token)
    }
    
    pub fn build_create_changeset(&self, step_name: &str) -> (CreateChangeSetInput, TokenInfo) {
        let token = self.context.derive_token_for_step(step_name);
        
        let builder = self.context.client.create_change_set()
            .stack_name(self.stack_args.stack_name.as_ref().unwrap())
            .client_token(&token.value) // Different field name for changesets!
            .change_set_name(&format!("iidy-{}", &token.value[..8]));
            
        (builder, token)
    }
}
```

### ConsoleReporter for Token Visibility
```rust
pub struct ConsoleReporter {
    show_tokens: bool,
    operation_name: String,
}

impl ConsoleReporter {
    pub fn show_primary_token(&self, token: &TokenInfo) {
        if !self.show_tokens { return; }
        
        match &token.source {
            TokenSource::UserProvided => {
                println!("🔑 Using provided idempotency token: {}", token.value);
            },
            TokenSource::AutoGenerated => {
                println!("🎲 Generated idempotency token: {} (save this for retries)", token.value);
            },
            TokenSource::Derived { from, step } => {
                println!("🔗 Using derived token: {} (from {} for step {})", token.value, from, step);
            },
        }
    }
    
    pub fn show_step_token(&self, step: &str, token: &TokenInfo) {
        if !self.show_tokens { return; }
        
        match &token.source {
            TokenSource::Derived { from, .. } => {
                println!("   🔄 Step '{}' token: {} (derived from {})", step, token.value, &from[..8]);
            },
            _ => {
                println!("   🔄 Step '{}' token: {}", step, token.value);
            },
        }
    }
}
```

## Multi-Step Operation Example

### Update Stack with Changeset
```rust
pub async fn update_stack_with_changeset(
    opts: &NormalizedAwsOpts,
    args: &UpdateStackArgs,
    stack_args: &StackArgs,
) -> Result<()> {
    // 1. Setup context and reporting
    let config = aws::config_from_opts(opts).await?;
    let client = Client::new(&config);
    let time_provider: Arc<dyn TimeProvider> = Arc::new(ReliableTimeProvider::new());
    let context = CfnContext::new(client, time_provider, opts.client_request_token.clone()).await?;
    
    let reporter = ConsoleReporter::new("update-stack --changeset", true);
    let builder = CfnRequestBuilder::new(&context, stack_args);
    
    // 2. Show primary token
    reporter.show_primary_token(&context.token_info);
    
    // 3. Step 1: Create changeset
    let (create_builder, create_token) = builder.build_create_changeset("create-changeset");
    reporter.show_step_token("create-changeset", &create_token);
    
    println!("🔄 Creating changeset...");
    let changeset_response = create_builder.send().await?;
    let changeset_id = changeset_response.id().unwrap();
    println!("✅ Changeset created: {}", changeset_id);
    
    // 4. Step 2: Execute changeset
    let (execute_builder, execute_token) = builder.build_execute_changeset(changeset_id, "execute-changeset");
    reporter.show_step_token("execute-changeset", &execute_token);
    
    println!("🚀 Executing changeset...");
    execute_builder.send().await?;
    
    // 5. Step 3: Watch stack progress
    watch_stack_with_context(&context, stack_args.stack_name.as_ref().unwrap(), Duration::from_secs(5)).await?;
    
    // 6. Show summary
    reporter.show_operation_summary(&context);
    
    Ok(())
}
```

```rust
pub async fn make_cfn_api(opts: &NormalizedAwsOpts) -> Result<Arc<dyn CloudFormationApi>> {
    if let Some(ref fixture_set) = opts.fixture_set {
        Ok(Arc::new(MockCloudFormationApi::new(fixture_set.clone())))
    } else {
        let config = aws::config_from_opts(opts).await?;
        let client = Client::new(&config);
        let time_provider = Arc::new(ReliableTimeProvider::new());
        let ctx = CfnContext::new(client, time_provider, opts.client_request_token.clone()).await?;
        Ok(Arc::new(ctx))
    }
}
```

## AWS API Field Name Inconsistencies

### Standard Operations
- `CreateStack`: `ClientRequestToken`
- `UpdateStack`: `ClientRequestToken`
- `DeleteStack`: `ClientRequestToken`
- `ExecuteChangeSet`: `ClientRequestToken`

### Changeset Operations
- `CreateChangeSet`: `ClientToken` (different field name!)

The `CfnRequestBuilder` handles these inconsistencies transparently.

## Benefits Summary

1. **🔄 Deterministic**: Same primary token always generates same sub-tokens
2. **🔒 Retry-safe**: Re-running commands with same primary token uses same sub-tokens
3. **📋 Traceable**: Can correlate all operations back to primary token
4. **🎯 Unique**: Each step gets distinct token for proper AWS idempotency
5. **👁️ Visible**: Full console output of token usage with source indication
6. **🧪 Testable**: Complete offline testing with fixture-based responses
7. **⚡ Fast**: No network dependencies in tests
8. **🛡️ Safe**: Centralized token generation prevents accidental reuse

2. **Incremental Testing**: Each phase has comprehensive test coverage
3. **Rollback Safety**: Each commit is independently functional

## Key Design Decisions

### Why Not StackArgs for Tokens?
- **Separation of concerns**: AWS operational settings vs CloudFormation template settings
- **User clarity**: Tokens are about retry safety, not template configuration
- **Cleaner precedence**: No need for complex CLI vs file precedence rules

### Why Deterministic Token Derivation?
- **Retry safety**: Same command with same primary token always uses same sub-tokens
- **Debugging**: Clear correlation between primary and derived tokens
- **Testability**: Deterministic behavior enables reliable testing

### Why Trait-Based Testing?
- **Current best practice**: Rust ecosystem standard for AWS SDK testing
- **Type safety**: Compile-time verification of mock implementations
- **Flexibility**: Can easily add new fixture scenarios
- **Performance**: Faster than HTTP-level mocking

This design addresses the fundamental limitations identified in the JavaScript implementation while leveraging Rust's type system for enhanced safety and testability.
